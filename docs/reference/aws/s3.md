# AWS S3

High-level S3 object operations over the `s3` client from
[`aws/clients`](./clients.md): plain-JS parameters in and out, AWS SDK v3
commands constructed internally. This is the abstraction boundary the library
commits to — `aws/clients` provisions the raw SDK client; `aws/s3` is the only
place that builds SDK commands against it, so no consumer (library or script)
ever imports `@aws-sdk/client-s3` command classes directly.

> **Status:** implemented and reviewed (ADR-0033). All 7 operations are real
> AWS SDK v3 dispatch over the `s3` client; 36 tests, 100% statement/branch/
> function/line coverage on `operations.ts`/`error.ts`.

## Origin

Surfaced while starting the `s3-objects` W3 consumer script: the roadmap
assumed the existing `s3` getter (a raw `S3Client`) was sufficient for a thin
op-dispatch script, the same way `dynamodb-crud` reached for
`dynamoDBDocument`. It isn't — dispatching a single S3 operation requires
constructing the matching `@aws-sdk/client-s3` command object, which
ADR-0027/ADR-0029 forbid inside `scripts/*/src`. See
[ADR-0033](../../adr/0033-aws-s3-operations-wrapper.md) for the full decision
record.

## Public API

Exported from `@m3l-automation/m3l-common/aws` (and re-exported under the
`AWS` namespace):

| Export                | Kind      | Summary                                                                  |
| --------------------- | --------- | ------------------------------------------------------------------------ |
| `listObjects`         | function  | Paginated object listing under an optional prefix, yielding pages.       |
| `headObject`          | function  | Fetch an object's metadata without downloading its body.                 |
| `getObject`           | function  | Download an object's full body and metadata.                             |
| `putObject`           | function  | Write (create or replace) an object.                                     |
| `copyObject`          | function  | Copy an object, within or across buckets.                                |
| `deleteObject`        | function  | Delete a single object by key.                                           |
| `deleteObjects`       | function  | Delete up to 1000 objects (S3's own `DeleteObjects` cap) in one request. |
| `S3ObjectSummary`     | interface | `{ key, size, lastModified, eTag }` — one entry from `listObjects`.      |
| `ListObjectsOptions`  | interface | `{ prefix?, pageSize? }` — `listObjects` parameters.                     |
| `S3Page`              | interface | `{ objects, nextContinuationToken }` — one page from `listObjects`.      |
| `S3ObjectMetadata`    | interface | `{ contentLength, contentType, eTag, lastModified }`.                    |
| `GetObjectResult`     | interface | `{ body, metadata }` — `getObject`'s return shape.                       |
| `PutObjectOptions`    | interface | `{ contentType? }` — `putObject` parameters.                             |
| `CopyObjectSource`    | interface | `{ bucket, key }` — the source object for `copyObject`.                  |
| `S3DeleteError`       | interface | `{ key, message }` — one per-key failure from `deleteObjects`.           |
| `DeleteObjectsResult` | interface | `{ deleted, errors }` — `deleteObjects`'s return shape.                  |
| `M3LS3OperationError` | class     | Typed error (`code: "ERR_S3_OPERATION"`) for any SDK rejection.          |

### Design choices

- **Pages, not objects, from `listObjects`.** Yields
  `AsyncGenerator<S3Page>` — `{ objects, nextContinuationToken }` — rather
  than individual objects, mirroring `aws/dynamodb`'s `queryItems`/
  `scanSegment`: a caller (e.g. `s3-objects`) can checkpoint on
  `nextContinuationToken` between pages without buffering the whole listing.
- **No retry inside `deleteObjects`.** Returns `errors` (per-key SDK
  failures) rather than retrying internally — retry policy stays the caller's
  concern via `Core.M3LRetryRunner`, consistent with `aws/dynamodb`'s
  `batchWriteItems`/`batchDeleteItems` not retrying `UnprocessedItems`
  internally.
- **1000-key cap enforced by `deleteObjects`.** Chunking a larger key set
  into 1000-key batches is the caller's job (mirrors S3's own `DeleteObjects`
  limit); passing more than 1000 throws `M3LS3OperationError`.
- **`copyObject` takes an explicit `source` bucket/key pair**, not a combined
  `Bucket/Key` copy-source string — avoids callers hand-building (and
  URL-encoding) the SDK's `CopySource` string themselves.
- **`headObject` returns `undefined` only on the modeled `NotFound` SDK
  exception name**, not on any SDK rejection that happens to carry a bare
  HTTP 404 status. An earlier draft also matched on `$metadata.httpStatusCode
=== 404`, but that over-matched: a misconfigured/nonexistent bucket
  (`NoSuchBucket`, also reported as 404) would have been silently read as
  "object doesn't exist" instead of surfacing as the configuration error it
  is. Narrowed to `name === "NotFound"` only (silent-failure review finding).

### `M3LS3OperationError`

Thrown by every function above when the underlying AWS SDK command rejects
(chained via `cause`), or when a function-level precondition is violated
(e.g. the 1000-key batch cap). Callers narrow via `code === "ERR_S3_OPERATION"`.

## Open questions / deferred follow-ups

- **Resolved:** `headObject` returns `undefined` on a `NotFound` SDK
  exception rather than throwing `M3LS3OperationError`, so `s3-objects`'
  `describe` op can distinguish "object doesn't exist" from "SDK call
  failed" without a try/catch of its own. See §Design choices above.
- Whether `listObjects` needs a `delimiter` option (folder-style listing) —
  deferred until `s3-objects`' contract shows a concrete need.
