## What didn't go as planned, and why

### 1. The waiter method's contract was internally contradictory as first drafted

The initial `docs/reference/aws/ecs.md` said `waitUntilServicesStable`
"wraps the SDK's own waiter" **and** that "a timeout or non-stable outcome
resolves ... rather than throwing." A `spec-conformance-reviewer`
contract-extraction pass (dispatched before `test-author`/`code-implementer`,
per the pipeline's Phase 1) caught that this is not achievable as written: the
SDK's current, non-deprecated `waitUntilServicesStable` waiter actually
_throws_ on any non-`SUCCESS` terminal state via its internal
`checkExceptions`, and its `FAILURE` terminal state surfaces as a plain,
unnamed `Error` — indistinguishable by identity from a genuine
`DescribeServices` call failure. The extraction pass also flagged that
`maxWaitTime` (required by the SDK's `WaiterConfiguration`) had no documented
default, and that the SDK's 5-value `WaiterState` enum had no mapping onto the
3-value `M3LECSWaiterResult.state` union.

**Why it happened:** The doc was drafted from the _intent_ ("timeout is data,
not a fault," mirroring `M3LAthenaClient.awaitResults`) without first reading
the installed SDK's actual waiter implementation to confirm that intent was
achievable with the chosen waiter function.

**Fix for future:** Before writing a doc-first behavioral contract for any
wrapped SDK operation whose resolve/throw semantics aren't a simple
`.send()` call (a waiter, a paginator, a multi-step poll), read the SDK's own
source/dist-types for that operation first and write the contract to match
what's actually implementable — not the other way around. Resolve any
resulting ambiguity (a default value, an unclassifiable-state fallback) in the
doc _before_ dispatching `test-author`/`code-implementer`, so both spokes work
from one settled contract instead of guessing independently and diverging.

### 2. A fix round required a mini TDD loop of its own

Applying `silent-failure-hunter`'s should-fix ("`createService`/
`updateService`/`deleteService` should throw, not silently default, when the
SDK response's `.service` field is absent") broke exactly 3 pre-existing tests
that had been written against the old, defaulting behavior.

**Why it happened:** A fix round is new writer code with no test between it
and the next state — the 3 broken tests were an expected, predictable side
effect of tightening the contract, not a defect in the fix itself.

**Fix for future:** When dispatching a fix-round `code-implementer`, explicitly
ask it to report which existing tests will newly fail (and why) rather than
have it also patch those tests — this keeps the writer/reviewer-adjacent
separation intact (the same spoke that tightens behavior shouldn't also
relax the tests that catch it) and gives the hub a checkable prediction to
verify before dispatching a separate `test-author` pass to update just those
tests. A single-reviewer focused confirmation re-review (not a full 4-way
fan-out) is sufficient to close the loop, matching the skill's existing
"scoped to the changed files only" guidance.

## Lessons learned

- **Read the SDK before writing a resolve/throw contract for it.** For any
  wrapped operation beyond a plain `.send()` call — a waiter, a paginator, a
  multi-step poll — verify the underlying SDK's actual throw/resolve behavior
  against its installed dist-types before drafting the doc's behavioral
  contract. An intent-first doc ("timeout is data, not a fault") can be
  unachievable with the chosen SDK primitive; catching this via a
  `spec-conformance-reviewer` contract-extraction pass _before_ RED/GREEN saved
  a guaranteed divergence between independently-guessing spokes.
  _(promoted → .claude/skills/implementing-submodules/SKILL.md)_

- **A scaffold placeholder needs `return Promise.reject(new X(...))`, not
  `throw new X(...)`, to satisfy both `require-await` and `.rejects` test
  semantics.** A non-`async` placeholder method that throws synchronously
  crashes the moment a test passes the call expression directly to
  `expect(fn()).rejects...` — the throw happens while evaluating the argument,
  before `expect`/`.rejects` ever gets a Promise to attach to. `Promise.reject`
  keeps the method non-`async` (no real `await` yet, satisfying the lint rule)
  while genuinely returning a rejected Promise, matching the eventual real
  `async` implementation's calling contract.
  _(promoted → .claude/skills/scaffolding-submodules/SKILL.md)_

- **The `expectTypeOf` runtime-invocation gotcha is not theoretical.** An
  `expectTypeOf(operations.method()).resolves.toEqualTypeOf<T>()` assertion
  still calls `method()` at runtime; against a placeholder that unconditionally
  rejects, the un-awaited promise surfaced as an actual "Unhandled Rejection"
  vitest error, not just a normal test failure. Fixed with a pure type-level
  assertion that never invokes the method:
  `expectTypeOf<Awaited<ReturnType<T["method"]>>>().toEqualTypeOf<U>()`. Worth
  reaching for this pattern by default whenever type-testing a fallible async
  method against a not-yet-implemented (or otherwise unmocked) target.

- **Verify a sibling module's field-naming convention against the SDK, don't
  assume it transfers.** `@aws-sdk/client-ecs` uses camelCase throughout
  (`cluster`, `serviceName`, `desiredCount`) where the sibling `client-lambda`
  wrapper it was modeled on uses PascalCase (`FunctionName`). Checking the
  installed package's dist-types directly (rather than pattern-matching the
  closest sibling's code shape) also surfaced the plural-array-vs-singular
  response nesting split (`DescribeServices`/`DescribeClusters` →
  `.services[0]`/`.clusters[0]`; `Create/Update/DeleteService` → singular
  `.service`) — flagged in advance as the most likely mapping bug, and it
  never materialized in the implementation.

- **A fix round's blast radius on existing tests is predictable — ask the
  writer spoke to report it, not fix it.** Tightening a should-fix (resolve →
  throw) breaks any test written against the looser prior behavior. Keeping
  `code-implementer` scoped to `src/` only and routing the predicted test
  breakage to a separate `test-author` dispatch kept the fix round auditable
  and the writer/test separation intact, at the cost of one extra
  (short) spoke round-trip.

- **Provenance sidecars for a new doc page are hand-authored, not
  generated.** `bin/check-doc-provenance.mjs` only re-verifies
  `*.provenance.json` files already on disk (`readdirSync` over existing
  sidecars) — there is no `gen:provenance`-style generator for a brand-new
  one. Computed blob hashes directly via `git hash-object <file>` (matching
  the checker's own internal `hashBlobs` helper) and modeled the new sidecar's
  section/heading structure on the closest sibling (`lambda.provenance.json`).
