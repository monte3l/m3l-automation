# Work log — `core/text` submodule (2026-07-02)

This log covers implementing the `core/text` multi-format text-extraction
submodule end-to-end through the `implementing-submodules` hub-and-spoke
pipeline: a Phase-1 re-spec to reconcile the module's six backing libraries with
the non-negotiable minimal-runtime-dependencies constraint, then TDD (RED →
GREEN), a five-spoke review, and the `/syncing-docs` reconciliation. It records
what shipped, what matched the plan, the divergences (several spoke truncations,
a post-review rule slip, and two real review findings), and the durable lessons.

Plan of record: [`docs/plans/text-submodule-implementation.md`](../plans/text-submodule-implementation.md)

## Summary

`core/text` is the **10th of 22** submodules, surfaced through the `Core`
namespace barrel — the three-entry `exports` map is unchanged, so this is a
**minor** (`feat`). Twelve public symbols: `M3LTextExtractorRegistry`,
`M3LPlainTextExtractor`, the five optional extractors (`M3LPdfTextExtractor`,
`M3LDocxTextExtractor`, `M3LXlsxTextExtractor`, `M3LEmailTextExtractor`,
`M3LZipTextExtractor`), `ZIP_DEPTH_SYMBOL`, the `M3LTextExtractor` interface, the
`M3LTextExtractionOptions`/`M3LTextExtractionResult` types, and
`M3LTextExtractionError`.

- **Dependency posture (re-spec decision).** Registry + `M3LPlainTextExtractor`
  are dep-free core (Node `fs`); the five library-backed extractors are opt-in.
  Their libraries (`unpdf`, `mammoth`, `read-excel-file`, `mailparser`,
  `cheerio`, `adm-zip`) are declared as **optional `peerDependencies`**
  (`peerDependenciesMeta.*.optional`) + `devDependencies` for tests, and loaded
  via a lazy dynamic `import()` on first `extract()`. Base consumer install pulls
  in none of them. `@types/adm-zip` + `@types/mailparser` added; `mammoth` has no
  types, so a minimal ambient shim lives at `src/internal/types/mammoth.d.ts`.
- **Tests:** 53 in `tests/text.test.ts` (34 RED → +15 coverage → +4 regression);
  full suite 936 passing. Happy paths use real fixtures + real libs; lazy-load
  and absent-library paths mock the dynamic import.
- **Coverage:** every `text` src file ≥ 80% (docx/pdf/xlsx/zip at 100%) after
  removing unreachable defensive branches.
- **Gates:** `build`, `typecheck`, `lint`, `check:scaffold`, `check:exports`
  (esm-only), `knip`, `check:provenance`, `check:doc-exports`,
  `check:impl-counts`, `check:index`, `lint:md` — all green.
- **Review verdicts (5 spokes):** spec-conformance **conformant** (no drift, no
  leaked exports); code-reviewer **approve with minor**; type-design **1
  must-fix** (narrow error `code`); silent-failure **FAIL** (1 HIGH + 1 MED);
  security **no must-fix** (S1 depth-clamp + S2 breadth/size caps —
  `maxEntries`/`maxTotalBytes` — both shipped in `zip.ts`). All
  must-fixes applied and re-verified.

Commits: `docs(text)` re-spec → `chore(deps)` → `feat(text)` implementation →
`docs(text)` doc reconciliation (+ a small `refactor(text)` un-exporting the
error option/code types).

## What went as planned

- **RED failed for the right reason** — `Cannot find module '../src/core/text/index.js'`,
  not a defect in the test logic.
- **The optional-peer-dependency posture worked exactly as designed** — the base
  install stays minimal, `check:exports`/`knip`/tree-shakeability all clean, and
  the C6/C7 tests prove construction + registration import nothing until
  `extract()` runs.
- **spec-conformance stayed clean twice** — the Phase-1 re-spec review produced
  the TDD contract, and the post-implementation conformance pass found the 12
  symbols exact, no extras, no internal helper leaked through the barrel.
- **`/syncing-docs` ran end-to-end without a manual count edit list** — it
  derived the 9 → 10 bump and named every stale site; the numerator was never
  hand-tracked.

## What didn't go as planned, and why

### 1. The implementer spoke truncated repeatedly on the token-heavy GREEN phase

The GREEN implementer returned a mid-sentence, non-summary message three times
(initial GREEN, the first fix batch, the second), each time having done real
work but hit its turn limit before reporting. Every time, the hub verified state
directly (listed files, grepped the barrel, re-ran typecheck/lint/test) and
resumed with a scoped punch-list rather than trusting the truncated return.

**Why it happened:** a six-extractor module with per-library interop is
token-heavy, and bounded-I/O rework (lint/type fixes across 10 files) compounds
it — the single "make it all pass" dispatch exceeds one turn.

**Fix for future:** for a multi-class module, split GREEN into smaller scoped
dispatches (e.g. registry+core first, then extractors, then lint cleanup) and
always verify the writer's state on disk before believing a summary — a
truncated return is the norm here, not the exception.

### 2. The plan's implemented-count premise was stale

The plan said "5 of 22 implemented / text is the 6th"; the live status file
showed 9 of 22, making text the 10th. It changed nothing material but would have
mis-stamped prose if trusted.

**Why it happened:** the plan was authored several submodules earlier; counts
rot between authoring and execution.

**Fix for future:** re-validate a stored plan's counts and "what exists"
premises against `docs/implementation-status.md` before acting, and delegate the
actual count reconciliation to `/syncing-docs` (which owns the site list).

### 3. The spec asserted an untestable contract ("first-registered wins") with no registration API

The re-spec review (spec-conformance) flagged that the page relied on "the first
registered extractor" while defining no registration surface, no
`M3LTextExtractor` interface members, and no no-match behavior — blocking a
testable contract. These were resolved in the re-spec (declarative
`mimeTypes`/`extensions` arrays, optional-ctor-array + `register()`, no-match
throws) via a user decision before RED.

**Why it happened:** the original spec described behavior in prose without
pinning the mechanism that makes it observable.

**Fix for future:** when re-speccing, close every untestable assertion (how does
registration happen, what does the no-match / empty case do) in the same pass —
not just the change that motivated the re-spec (the dependency posture).

### 4. Library interop shapes (CJS `export =`, subpaths, missing types) caused a cluster of type/lint errors

`adm-zip` is CJS `export = AdmZip` (dynamic-import default didn't resolve under
NodeNext); `read-excel-file` exposes no bare entry so the loader imports
`read-excel-file/node`; `mammoth` ships no types and has no `@types/mammoth`;
`unpdf`'s PDF proxy type resolves as an "error type" to the type-aware linter.
Each surfaced as a separate typecheck/lint failure across GREEN iterations.

**Why it happened:** the six libraries have heterogeneous module shapes, and the
dependency gate approved them by name without recording their interop shape.

**Fix for future:** at the dependency gate, record each lib's module shape
(`export =` vs ESM, bare vs subpath entry, bundled types vs `@types` vs none) so
the implementer front-loads the ambient shim / interop cast / subpath specifier
instead of discovering each mid-GREEN.

### 5. Tightening the result type to `readonly` broke an `expectTypeOf` object-match

Making `M3LTextExtractionResult` fields `readonly` (a type-design should-fix)
broke `expectTypeOf<…>().toMatchObjectType<{ text; truncated }>()` with TS2554,
because the matcher compares mutability. A test-author follow-up updated the
expected shape to `readonly`.

**Why it happened:** the type-test's expected literal was mutable; an
object-match matcher treats a `readonly` implementation as not matching.

**Fix for future:** this is already captured in `.claude/rules/tests.md` — when
an implementation type is (correctly) `readonly`, the assertion's expected
literal must be `readonly` too. The run re-confirmed the rule; no new promotion
needed.

### 6. Per-file coverage failed on failure-path branches and unreachable defensive code

Happy-path-only tests left the extractor `catch` branches and untested loaders
below the 80% branch gate. Adding failure-path tests lifted statements, but three
files stayed pinned by genuinely unreachable defensive code — an
`if (cause instanceof M3LTextExtractionError) throw cause;` guard in docx/pdf/xlsx
(where the loader runs outside the `try`, so nothing inside ever throws that
type) and dead `bigint`/object arms in xlsx `formatCell`. The test-author
correctly declined to contort tests to hit dead code and routed it to the hub;
the implementer removed the dead branches (keeping the reachable one in zip.ts).

**Why it happened:** a defensive re-throw guard copied across extractors is only
reachable where the module re-dispatches (zip); everywhere else it is dead. And
`formatCell` was typed `unknown`, forcing an exhaustiveness fallthrough the real
cell union never reaches.

**Fix for future:** when an extractor's lazy loader is called _outside_ the
`try`, do not add the `instanceof <own error>` re-throw guard inside the catch —
it is dead. Type boundary helpers to the library's real value union rather than
`unknown` so no unreachable fallthrough arm is needed.

### 7. The ZIP re-dispatch swallowed genuine nested failures (silent-failure HIGH)

`#dispatchEntry` caught `M3LTextExtractionError` by **class** and returned
`undefined`, but that class carries three codes — a corrupt nested entry
(`ERR_TEXT_EXTRACTION`) or a missing peer dep for a _supported_ nested format
(`…_MISSING_DEP`) was silently dropped, not just a truly unsupported entry. Fixed
to skip only `ERR_TEXT_EXTRACTION_UNSUPPORTED` and re-throw the rest (+ a
corrupt-nested-entry regression test). This depended on the type-design must-fix
that narrowed `code` to a literal union, which made the discrimination
exhaustive.

**Why it happened:** the catch discriminated on the error _class_ when the
signal that distinguishes "benign skip" from "real failure" is the error _code_.

**Fix for future:** when one error class carries multiple `code`s, discriminate a
swallow/skip on the specific `code`, never on the class — swallowing the class
drops the failures the code was meant to distinguish.

### 8. "Depth of 2" prose vs. the implemented one-level cap

The spec said "limited to a default depth of 2"; the guard permits exactly one
level of nested descent. Reconciled by clarifying the spec: the cap counts total
archive layers including the root, so 2 = root + one nested level, and it bounds
recursion, not the initial open.

**Why it happened:** "depth" was ambiguous between 0-indexed nesting and
layer-count-including-root; prose and code each read it differently.

**Fix for future:** when a spec states a numeric bound, pin its exact off-by-one
semantics (what the counter counts, inclusive/exclusive) so prose, constant, and
tests cannot drift.

### 9. A caller-settable public symbol let a hostile depth bypass the cap (security S1)

`ZIP_DEPTH_SYMBOL` is a documented, caller-settable public export, so a negative
or non-finite value defeated the `depth + 1 >= cap` guard. Fixed by clamping the
read to a non-negative finite integer at the boundary (+ negative and non-finite
regression tests).

**Why it happened:** the depth was treated as trusted-internal, but the symbol is
part of the public surface, so its value is external input.

**Fix for future:** any caller-settable public option — even a symbol-keyed one
framed as "internal" — is external input and must be validated/clamped at the
public boundary.

### 10. A post-review edit re-introduced an exported error-options interface

The must-fix batch that narrowed the error `code` added
`export interface M3LTextExtractionErrorOptions` and
`export type M3LTextExtractionErrorCode` — landing _after_ the review spokes ran,
so no reviewer saw them. `.claude/rules/library-src.md` forbids exporting
error-constructor options interfaces. They were never barrel-exported (so public
surface stayed at 12 and conformance/knip stayed clean), but the file-level
`export` still broke the rule. Caught by the hub while grepping the rule during
work-log promotion; fixed by making both module-local.

**Why it happened:** fixes applied after the review fan-out are unreviewed by the
spokes; a rule violation in a late edit has no gate between it and the commit.

**Fix for future:** re-run at least a conformance/quality check (or a focused
grep against the rules) over changes introduced _after_ the review fan-out —
post-review fix batches are an unreviewed blind spot.

## Lessons learned

- **Verify the writer, don't trust the summary** — on token-heavy modules the
  implementer routinely truncates mid-work; list files, grep the barrel, and
  re-run the gates yourself, then resume with a scoped punch-list. This run
  confirms the existing `implementing-submodules` guidance; the new wrinkle is to
  split GREEN into smaller dispatches up front for multi-class modules.
- **Discriminate swallows by error `code`, not class** — when one `M3LError`
  subclass carries several codes, a `catch (e) { if (e instanceof X) skip }` drops
  the very failures the codes distinguish; narrow the skip to the specific
  benign code and re-throw the rest. _(promoted → `.claude/rules/library-src.md`)_
- **A dead `instanceof <own error>` re-throw guard signals the loader is outside
  the `try`** — copy-pasting the zip re-dispatch guard into non-recursive
  extractors leaves an unreachable branch that pins per-file branch coverage;
  omit it where nothing inside the try throws that type.
- **Caller-settable public options are external input** — validate/clamp them at
  the boundary even when they are symbol-keyed and framed as internal; exporting
  the key makes the value untrusted.
- **Record library module-shape at the dependency gate** — `export =` vs ESM,
  bare vs subpath entry, and bundled-types vs `@types` vs none each drive a
  concrete implementer action (interop cast, subpath specifier, ambient shim);
  capturing them up front avoids a per-lib GREEN failure cascade.
- **Re-speccing closes untestable assertions, not just the trigger** — resolve
  the registration surface / no-match / interface-member gaps in the same pass as
  the dependency posture, or the contract phase re-derives them.
- **Post-review fix batches are an unreviewed blind spot** — a rule violation
  introduced while applying must-fixes has no spoke between it and the commit;
  re-grep the rules (or re-run a focused conformance check) over late edits.
- **`readonly` implementation types need `readonly` expected literals in
  `expectTypeOf` object-matches** — re-confirmed; already in `.claude/rules/tests.md`.
