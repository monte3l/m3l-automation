## What didn't go as planned, and why

### 1. Retry composition had to move inside the wrapper, discovered only by reading classifier internals directly

The original scaffold-time assumption (and the Plan agent's initial design)
was that the wrapper would simply call
`new M3LRetryRunner(M3LPollingPolicies.sqsBatchSend())` around the raw SDK
send — a reasonable-looking default. Before finalizing the design, I read
`awsThrottlingClassifier`/`awsNetworkClassifier` directly rather than trust
the plan's framing, and found they read `.name`/`.code`/`.Code`/`$metadata`
straight off the thrown value, never `.cause`. That meant composing the
retry runner at the _caller_ side (a design also briefly considered) would
have silently broken throttling retry the moment the wrapper translated an
SDK error into `M3LSQSOperationError` before the runner ever saw it — a bug
that would have shipped clean (all happy-path tests pass) and only
surfaced under real AWS throttling in production.

**Why it happened:** the correct design depended on an implementation
detail of a shared library primitive (`core/polling/classifiers.ts`) that
isn't documented anywhere as a "must wrap the raw error" contract — it's an
implicit consequence of how the classifiers are written.

**Fix for future:** when a new module composes an existing retry/polling
primitive, read the classifier/policy's actual source before assuming a
composition point — don't infer the right layer from the primitive's public
API surface alone. This is a load-bearing correctness detail, not a style
choice, and it drove a real architectural decision (Zone A widening,
ADR-0026) that a shallower read would have missed entirely.

### 2. `joinBatchResult` silently dropped an orphaned `Failed[].Id` — caught by review, not by the original test suite

The RED-phase test suite (26 tests, contract-derived) never exercised the
case where an SDK `Failed[]` entry's `Id` matches no input entry (a
malformed/anomalous SDK response). `silent-failure-hunter` found it by
tracing the join-back algorithm's failure modes explicitly, and
`code-reviewer` independently converged on a related angle (untrusted
`Id: string | undefined`). Fixed by making `joinBatchResult` throw
`M3LSQSOperationError` on any orphaned entry, with a follow-up test added
by `test-author` in the review-fix round.

**Why it happened:** the contract phase (and the test cases it specified)
focused on the _documented, expected_ SDK response shapes — a `Failed[]`
entry always echoing a real input `Id` was an implicit assumption nobody
questioned until a reviewer specifically hunted for "what if the SDK
response is internally inconsistent."

**Fix for future:** when writing the test-case list for a batch/join-back
operation, explicitly include an "SDK response references something the
request never sent" case alongside the happy-path and documented-failure
cases — this class of bug (trusting a response field to always correlate
with the request) is a repeatable pattern worth a standing test-author
checklist item for any batch API. _(promoted → `.claude/agents/test-author.md`)_

### 3. The doc-count tooling had its own latent bug, exposed only by landing a 23rd module

`bin/lib/count-sites.mjs`'s `IMPLEMENTED_COUNT_SITES` patterns and
`buildImplementedListBlock`'s template both hardcoded the literal `22` for
the "of N" denominator — accurate the entire time the library sat at
exactly 22 submodules, but never actually _derived_. `gen:counts` correctly
recomputed the numerator (23 implemented) but left every "of 22" anchor
untouched, producing self-consistently wrong text ("23 of 22") across 6+
files. Fixed by converting those sites' `pattern` field to a factory
`(counts) => RegExp` deriving the denominator from `counts.total`, plus a
one-time hand-fix of the files whose stale "of 22" anchor text the
now-fixed generator could no longer find-and-replace (a chicken-and-egg:
the corrected pattern searches for "of 23", which doesn't exist in a file
that still says "of 22").

**Why it happened:** the tooling was written and tested only against the
"total never changes" assumption that held from authoring until this
session — a latent bug with no trigger until the first new submodule
landed since the 22-count was established.

**Fix for future:** any generator/checker pair whose correctness depends on
a count that "hasn't changed yet" should be exercised with a synthetic
count bump in its own test suite, not just against the live repo's current
state — the live repo alone can't catch a bug that only manifests on the
_next_ change. Flagging as a candidate for `bin/lib/count-sites.mjs`'s own
test coverage if it doesn't already inject a synthetic `deriveCounts()`
fixture with `total` incremented.

## Lessons learned

- **Read classifier/policy internals before designing a composition
  point.** A retry/polling primitive's correct integration layer can be an
  implementation detail invisible from its public API — verify by reading
  the source, not by inferring from the type signature. This was the single
  highest-leverage check in the whole session; skipping it would have
  shipped a silent retry-classification bug.
- **Batch/join-back operations need an explicit "orphaned response entry"
  test case.** Any API that correlates a response array back to a request
  array by an id/key needs a test for "the response references something
  the request never sent" — this is a repeatable failure class, not
  specific to SQS. _(promoted → `.claude/agents/test-author.md`)_
- **A generator/checker pair's correctness under a changing count should be
  tested with a synthetic count, not only asserted against the live repo.**
  The "N of M" tooling passed every check for months because M never
  changed — the bug was invisible until this session forced M to change.
- **The adversarial security refute pass is worth its cost on `aws/**`
  surfaces.** It didn't find anything here, but it worked five concrete,
  falsifiable angles with code citations rather than a rubber-stamp — that
  rigor is what makes a clean verdict actually trustworthy.
- **An informed override of a documented default policy (W5/ADR-0021) is a
  legitimate outcome, not a process failure** — surface the default and its
  rationale once, clearly, and respect the user's explicit choice to
  override it rather than re-litigating. Two confirmations in this session
  were enough; a third would have been friction, not diligence.
- **Parallel spokes on a shared file pair (src + tests) work correctly**
  when each spoke's write scope is disjoint and explicitly stated in its
  dispatch prompt — `code-implementer` never touched `tests/sqs.test.ts`
  and `test-author` never touched `client.ts`/`types.ts`, so the two
  parallel dispatches needed no locking or sequencing, just clear scope
  boundaries up front.
