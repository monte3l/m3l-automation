# Work log — `s3-objects` consumer script (2026-07-18)

This log covers Stage 2 of the two-stage `s3-objects` effort — the consumer
script itself, built on top of the `aws/s3` typed operations wrapper shipped
in Stage 1 (ADR-0033, PR [#160](https://github.com/monte3l/m3l-automation/pull/160),
logged separately at [`2026-07-18-aws-s3.md`](./2026-07-18-aws-s3.md)). It ran
through the full `implementing-scripts` hub-and-spoke pipeline and records
what shipped, what matched the plan, what diverged, and the durable lessons
for the next script.

## Summary

Shipped the `s3-objects` consumer script: `list`/`describe`/`get`/`put`/
`copy`/`delete`/`delete-batch` op-dispatch over the `aws/s3` typed operations
wrapper, reached via `script.aws.clients.s3` (a raw `S3Client` — `aws/s3` is
free functions, not a class facade like `M3LSQSOperations`). Five
`src/steps/` modules (`list-objects`, `single-object-ops`, `delete-batch`,
`destructive-gate`, `run-s3-objects` orchestrator), a 12-parameter config
schema, and a destructive-gate covering `put`/`copy`/`delete`/`delete-batch`
(`yes`-bypass-and-warn, mirroring `sqs-etl`). 102 tests; `pnpm typecheck`,
`pnpm lint`, `pnpm knip`, `pnpm exec vitest run scripts/s3-objects`, and
`pnpm build` all clean; `pnpm check:script-scaffold` conformant. Review
verdicts: `code-reviewer` (not mergeable as-is → 2 Must-fix + 3 Should-fix, both
classes closed), `security-reviewer` (0 Must-fix, 1 Should-fix, closed),
`silent-failure-hunter` (2 Must-fix, closed) — all three independently
flagged the same root-cause bug (see divergence 3).

Skills used: `starting-work` (via the original session's isolation gate),
`scaffolding-scripts`, `implementing-scripts`, `syncing-docs`,
`writing-commits`, `writing-work-logs`.

## What went as planned

- **Zero new runtime dependencies.** The script only ever calls `AWS.*`
  functions from `aws/s3`; it never imports `@aws-sdk/*`, so the dependency
  gate was a no-op.
- **The `sqs-etl`/`dynamodb-crud` fleet conventions transferred cleanly.**
  The destructive-gate pattern (`yes`-bypass-and-warn), the per-operation
  cross-parameter guard table, the `steps/` module layout, and a
  script-specific `M3LError` code family all mapped onto `s3-objects` without
  needing a new pattern invented.
- **RED failed for the right reasons.** All four not-yet-existing step
  modules failed with `Cannot find module`; the two modules that already
  existed as scaffold placeholders (`run-s3-objects.ts`, `config.ts`) failed
  on real assertions against the documented contract, not test-logic bugs.
- **The multi-file RED/GREEN eslint-import-resolution trap was avoided by
  design** — per `implementing-scripts`' own guidance, RED and GREEN were
  committed together in one commit rather than attempting a separate RED
  commit that would trip `import-x/no-unresolved` on the pre-commit hook.

## What didn't go as planned, and why

### 1. The contract-extraction pass surfaced 5 real gaps in the hub's own draft doc

Before dispatching `test-author`, the hub ran `spec-conformance-reviewer` in
CONTRACT mode against the just-written `docs/reference/scripts/s3-objects.md`.
It returned five genuine ambiguities rather than just restating the doc: (B)
unspecified `describe`-on-missing-object behavior, (D) an unspecified
run-summary shape, (F) an unspecified `failed.jsonl` record shape (bare key
vs. `{key, message}`), (G) no error-code family named anywhere in the
contract, and (H) unspecified destructive-gate-decline behavior (soft-land
vs. propagate). The hub closed all five by amending the contract page with a
new "Behavioral contract" section pinning down explicit decisions, then
formatted and committed the settled doc, before `test-author` ever ran.

**Why it happened:** A first-draft contract page written by the hub in one
pass, without an independent adversarial read, under-specifies exactly the
decisions a test-author and a code-implementer would otherwise have to guess
divergently from each other.

**Fix for future:** Run the contract-extraction spoke in CONTRACT mode
against a hub-authored draft page as a matter of course for any script with
more than a couple of operations or cross-parameter requirements, and treat
its flagged ambiguities as blocking — resolve them in the doc before RED, not
by letting two downstream spokes each pick their own interpretation.
_(promoted → `.claude/skills/implementing-scripts/SKILL.md`)_

### 2. Two background spokes returned truncated final-summary text mid-task

`test-author`'s initial RED dispatch and `code-implementer`'s initial GREEN
dispatch both returned a mid-sentence fragment as their final visible text
(e.g. "Now `destructive-gate.test.ts`.") instead of a completion report, even
though both had actually finished (or nearly finished) real work per their
journal files. In the GREEN case, the truncation happened right as the
implementer was about to apply a subtly wrong fix — coercing optional string
config fields (`prefix`, `key`, etc.) with `?? ""` to satisfy
`exactOptionalPropertyTypes`, which would have silently changed
"no prefix filter" into "prefix filter on the empty string." The hub caught
this by reading the actual `tsc` error output directly rather than trusting
either the truncated text or a resumed summary, and redirected the resumed
spoke to the conditional-spread pattern `aws/s3/operations.ts` already
established, rather than accepting the semantically-wrong shortcut.

**Why it happened:** Long, exploration-heavy background turns can hit a
context/turn limit and stop mid-thought; the visible final message is not a
reliable signal of either completion or correctness in either direction.

**Fix for future:** This is the existing "verify state directly, resume the
same spoke, never trust a truncated fragment" practice
(`docs/contributing/subagent-context-management.md`) working as intended —
no new rule needed, just a confirming data point. Recorded here rather than
promoted, since the durable rule already exists.

### 3. Three independent review spokes converged on the identical bug; one found a second, subtler one

`code-reviewer`, `security-reviewer`, and `silent-failure-hunter` — dispatched
in parallel, with no visibility into each other's findings — all
independently flagged the exact same root cause: `single-object-ops.ts`'s
`describe`/`get`/`put` left local `readFile`/`writeFile` calls unwrapped,
throwing a raw Node `fs` error instead of the documented
`ERR_S3_OBJECTS_OUTPUT`-coded `M3LError`, inconsistent with the sibling steps
(`list-objects.ts`, `delete-batch.ts`) in the same diff, which both already
wrapped the identical failure class correctly. `silent-failure-hunter`
additionally found a second, more subtle bug the other two missed:
`delete-batch.ts` silently discarded already-accumulated `deleted`/`errors`
state when a later chunk's `AWS.deleteObjects` call rejected outright (a
fatal whole-call SDK failure, distinct from the normal per-key-errors
partial-failure path, which correctly aggregates and continues). Both were
fixed in one round; the fatal-chunk-failure fix pushed `runDeleteBatch` three
lines over the `max-lines-per-function` eslint cap, requiring one small
follow-up extraction dispatch.

**Why it happened:** The unwrapped-fs-error bug was a straightforward
inconsistency between sibling modules written in the same GREEN pass — an
easy miss for a single reviewer, but not for three independent ones. The
partial-progress-loss bug required tracing the fatal-vs-partial distinction
in `AWS.deleteObjects`'s actual contract, which only `silent-failure-hunter`'s
narrower error-handling focus caught.

**Fix for future:** When multiple independent review spokes converge on the
identical file:line finding, treat that convergence itself as a strong
confidence signal and prioritize it — it happened here (3/3) and both flagged
issues were real, reachable, and previously untested.

### 4. A follow-up test-author dispatch declined to codify an incorrect claim from the review-fix report

The review-fix implementer's report claimed `list-objects.ts`'s
local-write-failure path would surface as `ERR_S3_OBJECTS_OUTPUT`. Dispatched
to add the corresponding test, `test-author` read the actual
`M3LJSONListExporter`/`M3LWriteStreamLifecycle` source first and found that a
write failure from the JSON exporter always arrives **already wrapped** as an
`M3LError` (coded `ERR_JSON_LIST_EXPORT`) before it reaches
`list-objects.ts`'s own catch block — so the `ERR_S3_OBJECTS_OUTPUT` wrap
branch there is only reachable for a genuinely raw, non-`M3LError` cause,
which the real exporter never produces. `test-author` wrote the test against
the actually-observed behavior (`ERR_JSON_LIST_EXPORT`, propagated
unmodified) and flagged the discrepancy in its report instead of writing a
test that would have falsely codified the claimed-but-untrue behavior. This
is not a bug — it's the same "if already an `M3LError`, rethrow unmodified"
defensive guard used throughout the codebase, including `aws/s3/
operations.ts` itself.

**Why it happened:** A prior spoke's written report is a summary, not ground
truth; an assumption stated confidently in one report can still be wrong
relative to the real source.

**Fix for future:** A spoke asked to write a test against another spoke's
claimed behavior should verify that behavior against the real source before
writing the assertion, and report a discrepancy rather than silently
propagating an upstream assumption into a test that would then lock in the
wrong expectation.
_(promoted → `.claude/agents/test-author.md`)_

## Lessons learned

- **Run contract-extraction in CONTRACT mode against a hub-authored draft,
  before RED.** For any script with several operations or cross-parameter
  requirements, treat the contract page as a hypothesis to be adversarially
  reviewed, not a finished spec — it found 5 real gaps here that would
  otherwise have been guessed divergently downstream.
  _(promoted → `.claude/skills/implementing-scripts/SKILL.md`)_
- **Convergent independent review findings are a trust signal.** When 2+
  review spokes with no visibility into each other's output flag the same
  file:line issue, prioritize it — redundancy across reviewers is exactly
  what the multi-spoke review fan-out is for.
- **A truncated spoke summary is not evidence of success or failure —
  verify via the journal and independent execution either way.** Confirmed
  again here; the existing `subagent-context-management.md` practice
  already covers this, so no new rule was added.
- **Downstream spokes should verify, not just trust, an upstream spoke's
  claimed behavior when writing an assertion against it.** A report can be
  wrong; the source is the ground truth.
  _(promoted → `.claude/agents/test-author.md`)_
