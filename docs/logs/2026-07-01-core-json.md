## What didn't go as planned, and why

### 1. The plan being audited was built on a stale premise

The `/audit` target (`json-submodule-implementation.md`) claimed a pre-existing
doc-count inconsistency to reconcile: `README.md` at 3/22, `docs/README.md` at
2/22, and a `CLAUDE.md:501` "5 of 22" line. Verification against the live repo
showed all of these were wrong â€” `README.md` and `docs/README.md` already read
**5/22**, and `CLAUDE.md` has no implemented-count line at all. The plan also
**omitted** the `gen:index`/`check:index` reference-index step and **missed two
count-bearing files** (`packages/m3l-common/README.md` and `docs/index.html`)
that the real reconciliation must touch.

**Why it happened:** The plan was authored against an earlier repo state (or from
assumption) and never re-validated; the counts had since been corrected, and the
plan's edit list was hand-derived rather than sourced from `/sync-docs`, which
owns the authoritative list of count sites.

**Fix for future:** Audit any stored plan against live repo state before
executing it â€” treat every factual claim (counts, line numbers, file lists) as
possibly rotted. Delegate count reconciliation to `/sync-docs` rather than a
hand-written edit list.

### 2. The contract grew from ~8 to 10 exports mid-implementation

The doc listed 5 named symbols plus an unnamed "depth / options / result types"
cluster. Two review-driven, user-approved decisions added a dedicated
`M3LJSONFormatDetectionError` (for API consistency with `environment`/`utils`
error subclasses) and a branded `M3LConfidence` (to enforce the `[0,1]`
invariant the type left in prose). Both are public and had to be documented in
`json.md` and mapped in the provenance sidecar before they could pass
spec-conformance.

**Why it happened:** The spec under-specified the detection result types and the
error channel; reviewers correctly flagged the loose `confidence: number` and
the base-`M3LError`-with-string-code pattern as gaps worth closing pre-1.0.

**Fix for future:** When adding public types beyond the documented surface,
update the `.md` spec **and** the provenance sidecar in the same change set â€”
otherwise spec-conformance flags them as undocumented drift and provenance has no
heading to map them to.

### 3. Bounded-read rework required a coordinated test re-mock, and error-wrapping was incomplete

The detector originally read the whole file via `readFile` for every depth. The
security and code reviewers flagged this as a memory-DoS that defeats the
`shallow`/`standard` tiers' documented purpose. Reworking to bounded
`open()`/`FileHandle` reads (extension=0B, shallow=1B, standard=64KiB prefix,
deep=three 512B windows via `stat`+offsets) broke the tests, which mocked
`readFile` â€” the test-author needed a coordinated round to re-mock `open()` and
its handle. A follow-up review round then found two Must-fix gaps in the new I/O:
`read()`/`stat()` failures **after** a successful `open()` were not wrapped in
`M3LJSONFormatDetectionError` (raw Node errors leaked), and the unguarded
`finally { await handle.close() }` could let a `close()` failure shadow the real
error. A further round found `detectDeep` **triple-counted** `bytesInspected` on
files â‰¤512B (an 18-byte file reported 56) because the three windows read the same
bytes and were joined. All fixed (try/catch around the read path re-throwing
existing `M3LError` unchanged; best-effort close; a `dedupeWindows` helper) and
re-verified clean.

**Why it happened:** Changing the impl's underlying I/O primitive invalidates any
test that mocks the old primitive; and a first-pass I/O rework only wrapped the
`open()` call, not the subsequent reads, while the deep-sampling counts didn't
account for overlapping windows on small files.

**Fix for future:** When the implementation mocks an I/O primitive, treat an I/O
primitive change as a two-spoke change (implementer + test-author) from the
start. When wrapping a fallible async resource, wrap the **whole** use (open +
read + stat), make cleanup in `finally` best-effort so it can't shadow the real
error, and test the post-acquire failure path â€” not just acquisition.

### 4. `check:test-counts` blocked by pre-existing Windows false-positives

`/sync-docs` step 6 (`check:test-counts`) runs the full Vitest suite, which fails
on Windows with 27 failures in `environment.test.ts`, `utils.test.ts`, and
`bin/tests/guard-doc-path.test.ts` â€” all POSIX-path / monorepo-walk-up
assertions (e.g. "paths start with `/`"). `json.test.ts` is not among them
(59/59 green).

**Why it happened:** Known environment issue (see the `windows-prepush-false-positives`
memory): those tests assume POSIX path separators and fail on Windows; unrelated
to json.

**Fix for future:** On Windows, read `check:test-counts` failures per-file â€” a
failure in `environment`/`utils`/`guard-doc-path` is the known false-positive,
not a regression. The check passes on CI/Linux.

### 5. Implementer subagents truncated their final reports near turn limits

Three separate `submodule-implementer` dispatches ended with a cut-off final
message (e.g. "Now let's typecheck and update the barrel...") instead of a
completion report. In one case the truncation hid that the barrel re-export for
`M3LJSONFormatDetectionError` was missing and that `asConfidence` was leaking as
a public export.

**Why it happened:** The implementer ran long (bounded-I/O rework is
token-heavy) and stopped mid-work; the returned "final" text was a mid-thought,
not a summary.

**Fix for future:** After any long writer spoke, verify state directly â€” list the
created files, grep the barrel exports, run `typecheck`/`lint`/tests â€” rather
than trusting the returned report. Resume the spoke via `SendMessage` to finish
concrete remaining gaps.

## Lessons learned

- **Re-validate stored plans against the live repo before executing.** Counts,
  line numbers, and file lists in a saved plan rot; verify each claim and
  delegate count reconciliation to `/sync-docs`, which owns the authoritative
  site list (it caught two count files the hand-written plan missed).
- **Document behavioral additions into the spec first.** Any behavior or public
  type beyond the literal doc (prototype guard, numeric-key semantics, new error
  class, branded type) must land in the `.md` and the provenance sidecar in the
  same change set, or spec-conformance reads it as drift.
- **An I/O-primitive change is a two-spoke change.** If tests mock `readFile` and
  the impl moves to `open()`/`FileHandle`, plan the implementer and test-author
  rounds together; the mock target must track the primitive.
- **Wrap the whole fallible resource lifecycle, not just acquisition.** Wrap
  open + read + stat under one typed-error catch, make `finally` cleanup
  best-effort so a failing `close()` can't shadow the real error, and test the
  post-acquire failure path.
- **Beware overlap in windowed file sampling.** Sampling start/middle/end must
  de-duplicate overlapping ranges on small files, or byte/line counts inflate
  (an 18-byte file reported 56 bytes before the fix).
- **Verify writer-spoke state directly.** Long implementer runs can truncate
  their final report and silently omit a barrel line or leak an internal helper;
  grep the barrel and run the gates instead of trusting the summary, and resume
  the spoke to close specific gaps.
- **Reference-index regeneration is mandatory for a new submodule.** `gen:index`
  then `check:index` after wiring the barrel and writing the provenance sidecar â€”
  the original plan omitted this and CI's `check:index` would have failed.
