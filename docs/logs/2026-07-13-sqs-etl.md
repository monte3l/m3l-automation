# Work log — `sqs-etl` consumer script (2026-07-13)

This log covers the `sqs-etl` W2 consumer script (`dump/send/redrive/delete/
purge/transform` over Amazon SQS), resumed in `worktree m3l-automation-sqs-etl`
/ branch `feat/sqs-etl` after its blocking dependency — the `aws/sqs`
`M3LSQSOperations` library wrapper — merged as PR #119. It ran the full
`implementing-scripts` TDD pipeline (contract → RED → GREEN → review fan-out →
fix round → docs), and records what shipped, what diverged, and the durable
lessons for the next W2 script (`dynamo-crud`, `logs-insights` are in flight in
parallel).

## What didn't go as planned, and why

### 1. RED-phase tests for a brand-new script trip the pre-commit lint hook, unlike a scaffolded submodule

Committing the 9 RED test files (before any `src/steps/*.ts` existed) failed
`pre-commit`'s `eslint` stage with ~78 errors — `import-x/no-unresolved`
cascading into `@typescript-eslint/no-unsafe-*` because TS couldn't resolve
the not-yet-existing modules' types.

**Why it happened:** `scaffolding-submodules`'s pattern (a placeholder file per
planned export, each throwing a module-specific error) makes RED imports
resolve to a _real, wrong-behavior_ module, so lint stays clean. `scaffolding-
scripts` only lays down one starter step file (`run-sqs-etl.ts`), not one per
planned command — there was no scaffold-time hook laying down 8 placeholder
step files for `sqs-etl` specifically, since the six-command decomposition
wasn't decided until this session's contract-settling pass, well after
scaffolding.

**Fix for future:** For a multi-step consumer script (any W2+/W3+ script with
more than one planned step module), either (a) scaffold one throwing
placeholder file per planned step before dispatching `test-author`, mirroring
the submodule convention, or (b) accept committing RED tests and the GREEN
implementation together in one commit (what this session did) rather than
fighting the pre-commit hook for a separate RED commit. Option (b) is simpler
and was sufficient here; note it in `implementing-scripts` if this recurs on
`dynamo-crud`/`logs-insights` (5-step scripts each, per their own scaffolds).

### 2. A stale turbo build cache served library dist output missing the just-merged `aws/sqs` submodule

The first `pnpm build` after the rebase produced a `dist/aws/` directory
without an `sqs/` subfolder, and the test suite failed with `AWS
.M3LSQSOperationError is not a constructor` — not a real regression, but a
turbo cache hit replaying stale logs from before the worktree was rebased onto
merged `main`.

**Why it happened:** Turbo's shared-worktree cache had a build artifact from
this exact source-content hash cached from a _different_ worktree (the
`aws-sqs-operations` worktree itself, which built the identical `m3l-common`
source earlier). The cache hit was technically correct (same inputs, same
output) for that build, but this worktree's own `dist/` hadn't been populated
with it until the build actually ran once. A second `pnpm build` (no code
change) fixed it — the cache had already been warmed by the first invocation.

**Fix for future:** After rebasing a worktree onto a `main` that merged new
library submodules, run `pnpm build` once and verify `packages/m3l-common/
dist/aws/<new-module>/` actually exists before trusting downstream test
failures as real — a stale/first-warm cache read is a known false-positive
source, not just a "prepush-stale-dist-race" concern at push time.

### 3. `code-reviewer`'s Should-fix to reuse `M3LJSONListExporter` in `transform-records.ts` doesn't actually hold against the library's own type constraint

`code-reviewer` flagged `transform-records.ts`'s hand-rolled `fs.createWriteStream`
JSONL writer as duplicating what `M3LJSONListExporter` already provides,
noting all 4 sibling step modules use the shared exporter. The hub evaluated
this and did **not** apply it: `M3LJSONListExporter<TItem extends object>` is
generically bounded to `object`, but `transform-records`'s own contract (and
a passing test) requires writing raw non-object JSONL lines (a bare string,
a bare number) through unprojected when `fields`/`filters` are both empty —
something the exporter's type signature cannot represent. The hand-rolled
writer was a deliberate, load-bearing choice, not an oversight the reviewer
correctly caught the _shape_ of the duplication but not this constraint.

**Why it happened:** The reviewer read the 4 sibling modules (which write
`object`-typed messages/entries) and reasonably generalized, but didn't check
the exporter's generic bound against `transform-records`'s specific
non-object-passthrough contract, which is the one command mode that
genuinely can't use it.

**Fix for future:** When a review finding proposes reusing a shared library
primitive, verify the primitive's exact type signature against the specific
call site's contract before accepting the suggestion — a plausible
consistency argument can miss a real type constraint that makes the
"duplication" load-bearing rather than accidental. Document the rejection
inline (as this log and a corresponding comment do) rather than silently
declining it.

### 4. Two RED-phase test assertions searched for a substring that JSON-escaping made unreachable

`send-batch.test.ts`'s and `redrive-queue.test.ts`'s "per-entry failure
written to `failed.jsonl`" tests searched a written line's raw text for a
literal substring drawn from a value that was itself a JSON string nested
inside the outer `JSON.stringify`'d entry (e.g. searching for `'"id":2'` when
the actual written text contains the backslash-escaped `\"id\":2`, since the
`body` field's own JSON-shaped string gets its inner quotes escaped when the
whole entry is serialized). Both tests failed even though the implementation
was correct — verified by direct reproduction (`JSON.stringify` + `.includes`)
before touching either file.

**Why it happened:** The test author (a fresh RED-phase context, not yet
seeing the real serialized output) assumed a value's raw text would appear
unmodified in an outer JSON-serialized line, without accounting for the
outer `JSON.stringify` re-escaping the inner JSON-shaped string's quotes.

**Fix for future:** When a `failed.jsonl`/similar re-drive assertion needs to
locate a specific written stream by content-sniffing, search on a **top-level,
never-JSON-shaped** field of the outer object (an `id`, a `receiptHandle`) —
never a substring of a field whose own value happens to look like JSON. If
the value under test genuinely is JSON-shaped, assert via
`writtenJsonlRecords(stream)` structural equality instead of a raw substring
search.

### 5. Two writer spoke turns (test-author, and code-implementer twice) ran out of turn mid-task without reporting a truncated state

Across this session, `test-author`'s RED dispatch and `code-implementer`'s
first GREEN dispatch and its fix-round dispatch all returned final messages
that were clearly mid-thought ("Good. Now let's write the extended
`config.test.ts`.", "Now let's run the verification commands, starting with
vitest.", "Now redrive-queue.ts, apply similar extraction.") rather than
completion summaries. In every case the actual file-write progress (checked
via `git status`/journal file) was further along than the final message
implied — the spoke had done real, correct work and simply ran out of turn
before narrating it.

**Why it happened:** Each dispatch covered a large surface (9 test files;
9 step files + composition wiring; 4 files' worth of fix + hooks tests) for a
single subagent turn, and none of the three explicitly paced itself to leave
narration budget at the end.

**Fix for future:** This matches the documented `implementing-submodules`
guidance exactly ("verify the writer spoke's state directly — do not trust
its final report") and it held up in practice every time: `git status`/the
spoke's own journal file is the ground truth, and resuming the same spoke via
`SendMessage` (rather than re-dispatching fresh) got it to finish cleanly in
one case. No process change needed beyond continuing to apply this guidance —
worth reinforcing since it fired 3 times in one session, not a one-off.

## Lessons learned

- **Multi-step scripts need step-count-aware RED-commit planning.** A script
  decomposed into N step modules (N > 1) will trip the pre-commit lint hook on
  RED tests unless either scaffolded with N placeholder files up front, or RED
  and GREEN are committed together. json-etl (1 command) never hit this;
  sqs-etl (6 commands, 8 step modules) did. _(See divergence 1; promoted →
  .claude/skills/implementing-scripts/SKILL.md, Phase 2 RED.)_
- **A rebuild after rebasing onto a submodule-merging `main` is not optional.**
  Verify the new submodule's `dist/` output actually exists before trusting a
  "not a constructor"-style test failure as a real regression — it may be a
  stale/first-warm turbo cache read, not code drift. _(See divergence 2;
  promoted → .claude/skills/implementing-scripts/SKILL.md, Step 0.)_
- **Verify a reviewer's "reuse the shared primitive" suggestion against the
  primitive's actual type signature before accepting it.** A plausible
  cross-file consistency argument can overlook a generic bound (or other type
  constraint) that makes the flagged "duplication" load-bearing. _(See
  divergence 3.)_
- **Never raw-substring-search a written JSON line for a value that is itself
  JSON-shaped.** Nested `JSON.stringify` escapes the inner quotes, so the raw
  text never contains the unescaped substring — search on a top-level,
  never-JSON-shaped field instead, or assert structurally via the parsed
  record. _(See divergence 4.)_
- **Truncated-looking spoke completions are not a red flag on their own —
  verify, don't distrust.** Every one of the three long writer-spoke turns
  this session ended mid-narration but had done complete, correct work; `git
status` plus the spoke's own journal file settled it in seconds each time.
  _(See divergence 5.)_
- **Front-loading an archived-but-unexecuted plan before dispatching spokes is
  worth the read time.** The 2026-07-09 consumer-scripts implementation plan
  had already worked out the streaming/destructive-gate/retry conventions in
  detail; reading it before writing the contract page avoided any
  design-clarification round-trips with `test-author`.
