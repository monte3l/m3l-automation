# Work log — `core/json` extraction extension (2026-07-10)

This log covers W0-L1 of the consumer-fleet program — the first concrete unit,
extending `core/json` field-path extraction so downstream scripts can pull values
from variable, nested JSON. It ran through the `implementing-submodules` TDD
hub-and-spoke loop (spec-first → RED → GREEN → parallel review → doc
reconciliation) on branch `feat/json-extraction-paths`. It records what shipped,
what matched the plan, the divergences (a shared-primitive blast radius and a
provenance-drives-the-index gotcha), the deferred `onUnknownFormat` decision, and
the durable lessons.

Plan of record:
[`docs/plans/2026-07-09-consumer-scripts-implementation-plan.md`](../plans/2026-07-09-consumer-scripts-implementation-plan.md)

- **`navigateFieldPath`** gained **array indexing** on digit-only segments
  (`/^\d+$/`): `{items:["x","y"]}` at `items.1` → `"y"`. A numeric segment on a

## What didn't go as planned, and why

### 1. Widening a shared primitive broke a cross-module test outside the json suite

`navigateFieldPath` backs `M3LJSONListImporter`, so the array-indexing widening
changed the importer's behavior too: `importers.test.ts:884` ("a fieldPath
segment is never treated as an array index", asserting `[undefined]`) started
returning `["x"]`. The json module's own suite was fully green; the break only
appeared in the **full-suite** run the implementer did before handing back. It
was reconciled by `test-author` (flip the assertion + update a stale file-level
comment), and the importer's own spec page needed no change (it never promised
"no array indexing").

**Why it happened:** the old "arrays are never indexed" contract was encoded in
a consumer's test file, not in `json.test.ts`, so a module-scoped test run would
have missed it.

**Fix for future:** when widening a shared primitive, `grep` the **whole test
tree** (and every consumer's spec page) for the old contract before declaring
GREEN — never trust the changed module's own suite alone. Run the full suite,
not just the module's tests, as the GREEN gate for shared code.

### 2. The reference index is built from provenance sidecars, not the barrels

After committing, `gen:index` reported no change and `check:index` passed — yet
the json catalog entry still listed 10 symbols without `extractAll`. `buildIndex`
(`bin/lib/reference-index.mjs`) derives each module's `symbols` from its
`*.provenance.json` `sections[].sources[]`, **not** the source barrel. The
scoped provenance restamp (`--update --affected …`) had refreshed the sidecar's
`commit`/`retrieved` but never **added** `extractAll` to `sources[]`, so
`gen:index` faithfully reproduced 10 symbols and `check:index` passed vacuously
(committed == generated, both missing the new export). Adding `extractAll` to
both sidecar sections immediately tripped the `guard-index-staleness`
PostToolUse hook, and a re-`gen:index` lifted the count to 261.

**Why it happened:** two false signals lined up — `check:doc-exports` (which
walks the barrel) passed because the symbol _was_ documented in the `.md`, and
`check:index` passed because it only compares generated-vs-committed, not
generated-vs-barrel. Restamping is a timestamp operation, not a symbol-sync one.

**Fix for future:** a new public export must land in **three** places in the same
change set — the `.md` page, the provenance sidecar `sources[]`, and (via
`gen:index`) the reference index. Treat `--update` as timestamps-only; after it,
diff the sidecar's `sources[]` against the new barrel exports by hand. A
green `check:index` with no `gen:index` diff after adding an export is the tell
that the sidecar never learned about the symbol.

### 3. `check:api` tracks the exports map, not individual symbols

The plan anticipated an intentional `check:api` snapshot bump for the new
`extractAll` export. In fact `bin/check-exports-snapshot.mjs` snapshots the
three-entry `exports` map (`.`/`./core`/`./aws`), which is unchanged for any
barrel-surfaced symbol — so no snapshot update was needed.

**Why it happened:** a reasonable-but-wrong assumption in the plan about what
`check:api` covers.

**Fix for future:** a barrel-surfaced export needs no `check:api` action; that
gate only moves when a subpath in the `exports` map is added/removed/retyped.

## Lessons learned

- **Full-suite is the GREEN gate for shared primitives.** When a change widens a
  function other modules depend on, the module's own tests are insufficient —
  the old contract may be pinned in a consumer's test file. `grep` the whole
  test tree for the old behavior and run the full suite before GREEN.
- **A new export lands in three synchronized places.** `.md` page + provenance
  sidecar `sources[]` + regenerated reference index, all in one change set. The
  reference index derives symbols from the **sidecar**, not the barrel, so a
  scoped `--update` (timestamps-only) leaves the index silently short and
  `check:index` passes vacuously. _(reinforces `docs/logs/2026-07-01-core-json.md`
  divergence 2; promoted → `.claude/skills/syncing-docs/SKILL.md`)_
- **A no-op `gen:index` after adding an export is a red flag.** If `gen:index`
  produces no diff right after you added a public symbol, the sidecar hasn't been
  taught the symbol yet — check `sources[]` before trusting a green `check:index`.
- **Scoped provenance restamp via `--affected`.** `node bin/check-doc-provenance.mjs
--update --affected <srcfile>` restamps only the sidecars that reference that
  source file — the mechanism that honors the "don't touch the other 21
  sidecars" parallel-PR rule.
- **`check:api` is exports-map-scoped.** No snapshot bump for a barrel-surfaced
  symbol; it only moves on a subpath change.
- **Resolve extraction edge cases in the spec, not under review.** Pinning
  present-nullish, `*`-over-primitive, and digit-only-index semantics in
  `json.md` up front kept the contract producer, test-author, and implementer
  aligned and avoided a rework round.

## Deferred decision (library friction / follow-up)

The opt-in `onUnknownFormat: "throw" | "skip"` tolerant per-record import mode on
`M3LJSONListImporter` (plan §2 item 4) was **deferred**. W1 `json-etl` already
gets per-line JSONL tolerance from `importStream()`, so it is not on the W0 → W1
critical path. It was intentionally kept **out** of the spec page to avoid the
speculative-phrasing gate (`docs-consistency-reviewer` sweeps for
"not yet implemented / design target"); this log is its tracked home. Revisit
when a consumer actually needs per-record tolerance on irregular non-JSONL input.
