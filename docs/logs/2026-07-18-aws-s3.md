## What didn't go as planned, and why

### 1. The ROADMAP's W3 premise for `s3-objects` was wrong

The task began as "start work on `s3-objects`" per `docs/ROADMAP.md`'s W3
row, which described it as a thin op-dispatch script needing only "existing
getters." That's false: the `s3` getter is a raw `S3Client`, and
ADR-0027/ADR-0029 ban `@aws-sdk/client-s3` imports anywhere under
`scripts/*/src` â€” so a script cannot construct a single S3 command object,
meaning `s3-objects` actually needed this library wrapper built first. This
was caught during the `starting-work` gate's isolation-scope inference, before
a single line of script code was written.

**Why it happened:** the ROADMAP's W3 classification assumed every listed
service already had an ergonomic getter, without checking whether that getter
was a raw SDK client (needing a wrapper) or already a high-level operations
object (like `dynamoDBDocument`).

**Fix for future:** before starting a W3/W4-classified consumer script,
verify the target service's getter shape (`aws/clients/provider.ts`) actually
supports direct SDK command dispatch from a script â€” a raw client getter is a
signal the script is wrapper-gated, not W3-ready, regardless of what the
roadmap row says.

### 2. Two review spokes ran 30+ minutes with no completion; stopped and re-dispatched

`code-reviewer` and `silent-failure-hunter`'s first dispatch (part of a
5-agent parallel review fan-out over the same diff) ran 30+ minutes with no
completion, while the other 3 reviewers (security, spec-conformance,
type-design) finished in 104sâ€“186s each. The user flagged the asymmetry; the
harness confirmed both were still `status: running` (not crashed or errored),
which on its own couldn't distinguish thoroughness from a stall. Both were
stopped via `TaskStop` and re-dispatched with a more tightly bounded file
list; both completed in under 110 seconds.

**Why it happened:** unconfirmed. No transcript of the stopped runs was
inspected (the tooling explicitly warns against reading a live/killed agent's
raw `.output` JSONL directly â€” it would overflow context), so there is no
verified root cause. The retry's tighter scope is the only known difference
between the two attempts, but the original prompts already named specific
target files rather than a bare topic, and the three fast reviewers also read
comparison files without incident â€” which weakens a clean "unbounded scope
caused it" explanation. A transient infrastructure hiccup on the first
dispatch, independent of prompt wording, remains equally plausible. This is a
single occurrence; no general pattern should be inferred from it.

### 3. `headObject`'s not-found detection over-matched â€” a contract-pass suggestion that needed review-layer correction

The `spec-conformance-reviewer` contract-extraction pass recommended
detecting a missing S3 object via
`cause.name === "NotFound" || cause.$metadata?.httpStatusCode === 404`, and
`code-implementer` built exactly that. On the review fan-out,
`silent-failure-hunter` flagged it as a genuine HIGH-severity finding:
the `httpStatusCode === 404` fallback also matches
`NoSuchBucket` (a misconfigured/nonexistent bucket, also reported as HTTP
404), which would then be silently read as "the object doesn't exist" instead
of surfacing the real configuration error. Narrowed to `name === "NotFound"`
only; the existing not-found test (which set both fields) stayed green
without modification.

**Why it happened:** the contract pass reasoned from the AWS SDK's rejection
shape in isolation, without weighing the over-matching risk of a bare
HTTP-status fallback against sibling exceptions that share the same status
code for an unrelated reason.

**Fix for future:** treat a contract-extraction pass's suggested
_implementation detail_ (as opposed to its documented _export shape_) as a
hypothesis for the reviewers to stress-test, not a decision to implement
verbatim â€” this is precisely the redundancy the review fan-out (especially
`silent-failure-hunter`, whose job is exactly this class of over-broad-match
error handling) exists to provide.

### 4. Branch coverage fell short after the first GREEN pass; closed via a precise, line-numbered gap list

After the initial `code-implementer` pass, `operations.ts` sat at 60% branch
coverage against the project's 80% per-file gate â€” most of the gap was `??`
fallback branches (e.g. `response.ContentLength ?? 0`) and
`instanceof M3LS3OperationError` pass-through guards that the initial 21
tests never exercised (every mock always supplied every optional field, and
no test ever rejected `send` with a pre-built `M3LS3OperationError` to prove
the guard's pass-through path). Reading `coverage/coverage-final.json`'s
per-branch detail directly (not the `vitest --coverage` text-table summary,
which hides fully-covered files and shows no per-branch detail at all)
produced an exact, line-numbered list of the 13 missing branches, which
`test-author` closed in a single follow-up round â€” landing `operations.ts`
and `error.ts` at 100% on all four metrics.

**Why it happened:** the initial RED round wrote one happy-path + one
failure-path test per exported function (satisfying the "at least one pair"
scaffold bar), which is necessary but not sufficient for the fallback-branch
and pass-through-guard paths that only a targeted edge-case test exercises.

**Fix for future:** after the first GREEN pass on a module with multiple `??`
defaults or an `instanceof`-guarded catch, budget for a dedicated
coverage-completion round from the start rather than treating the initial RED
tests as sufficient â€” and always source the gap list from
`coverage/coverage-final.json`'s branch map, not the console summary table.

### 5. `syncing-docs` surfaced a real latent tooling bug, not just doc drift

Running the doc-reconciliation pipeline after flipping `aws/s3`'s status to
âś… produced a self-consistent but _wrong_ result: "26 of 27 submodules,"
omitting `s3` from the implemented-list prose entirely, even though its table
row correctly showed `Status: âś…`. Root cause:
`parseImplementationStatus()` (`bin/lib/reference-index.mjs`) and
`check-test-counts.mjs`'s header-row guard both used a
`/^[a-z][a-z-]*$/`-shaped regex to distinguish a real submodule-name row from
a heading/separator/prose row â€” neither character class included digits, so
`"s3"` (the first ADR-0028-compliant submodule name containing one) was
silently dropped from the status map, the generated implemented-list block,
the `catalog.json` per-module status, and the test-count check. Fixed both
regexes to `[a-z0-9-]`, made `parseImplementationStatus()` accept an optional
`content` param (mirroring `count-sites.mjs`'s injectable `getStatus`
pattern) so it's directly unit-testable, and added a regression test.
Re-running the full `sync:docs` pipeline after the fix correctly produced
"27 of 27" with `s3` included everywhere.

**Why it happened:** every submodule name until now (`errors`, `dynamodb`,
`cloudwatch-logs-insights`, `sqs`, `signing`, â€¦) happened to be pure
lowercase-and-hyphens, so a digit-excluding regex never manifested as a bug â€”
until ADR-0028's full-official-AWS-service-name convention produced one
(`s3`) that contains a digit.

**Fix for future:** this specific class of bug is now closed by the
regression test in `bin/tests/reference-index.test.ts`, but any _new_
name-matching regex added to `bin/**` for a "submodule name" or similar
identifier column should allow digits from the start â€” more digit-bearing
official AWS names are coming (e.g. a future `ec2` submodule).

## Lessons learned

- **A contract-extraction pass's suggested implementation detail is a
  hypothesis, not a decision.** The review fan-out â€” especially
  `silent-failure-hunter` â€” exists precisely to catch an over-broad-match
  error-handling suggestion that a contract pass reasoned into without
  weighing the false-positive risk against sibling error shapes.
- **Read `coverage/coverage-final.json`'s branch map directly when coverage
  falls short, not the text-table summary.** It hides fully-covered files and
  carries no per-branch detail, so it can't produce the precise,
  line-numbered gap list that converges a coverage-completion round in one
  pass instead of several. _(already captured in `.claude/rules/tests.md`;
  this run reconfirms it holds)_
- **Doc-tooling name-matching regexes must allow digits.** ADR-0028's
  full-official-AWS-service-name convention guarantees more digit-bearing
  submodule names are coming; any new `bin/**` regex that validates a
  "submodule name" column should be written against `[a-z0-9-]` from the
  start, not discovered as a bug when the first one lands.
