# New gated library module: typed SQS operations wrapper (`aws/sqs`)

**Status: shipped** (PR #119)

## Context

While scoping the `sqs-etl` consumer script's contract, `spec-conformance-reviewer`
surfaced that `AWSClientProvider`'s `sqs` getter — like every AWS getter in
`packages/m3l-common/src/aws/clients/provider.ts` — returns a **raw
`SQSClient`** by deliberate design. Building `sqs-etl` directly against it
would have forced the script to import `@aws-sdk/client-sqs` command classes
and take on that package as a direct dependency. `sqs-etl` is the first SQS
consumer, so the repo's own ADR-0021 D4/D5 gate and W5 "promotion pass"
convention would normally defer a library wrapper until a _second_ script
duplicated the pattern. The user was asked twice whether to follow that
default or override it; both times the answer was to build the wrapper
first — a deliberate, informed decision, not a process gap. `scripts/sqs-etl`
stayed scaffolded-but-paused on a separate branch/worktree until this module
landed.

## Approach / Decisions

- New AWS submodule `packages/m3l-common/src/aws/sqs/` — `M3LSQSOperations`,
  the first bespoke typed-wrapper-over-raw-SDK-client pattern in the codebase
  (`dynamoDBDocument` only wraps via the SDK's own `.from()`, not custom
  methods). Surfaces through the `aws/index.ts` namespace barrel; no
  `exports`-map change.
- Public methods: one-shot `receive()` (not a drain-loop generator — the
  drain/termination policy stays script-specific), `sendBatch()`,
  `deleteBatch()`, `purgeQueue()`. Plain boundary types
  (`M3LSQSReceivedMessage`, `M3LSQSSendEntry`, `M3LSQSBatchResult<T>`, etc.)
  keep `@aws-sdk` types off the public surface; `M3LSQSBatchFailure<T>` joins
  the SDK's `Failed[].Id` back to the original input entry internally.
- **Load-bearing design finding:** `awsThrottlingClassifier`/
  `awsNetworkClassifier` (`core/polling/classifiers.ts`) read
  `.name`/`.code`/`.Code`/`$metadata` directly off the thrown error, never
  `.cause` — so retry composition (`sendBatch`/`deleteBatch` via
  `M3LRetryRunner` + `M3LPollingPolicies.sqsBatchSend()`) had to live _inside_
  the wrapper, wrapping the raw `.send()` call before any
  `M3LSQSOperationError` translation. Composing retry at the caller side
  would have silently broken throttling retry.
- That requirement collided with ADR-0009's Zone A (`aws/**` may import only
  `core/errors`/`core/prompt`) — resolved by writing **ADR-0026**, which
  widens the exception list to add `core/polling` (verified acyclic:
  `core/polling` depends only on `internal/polling/*` and `core/events`,
  never `aws/`), alongside recording the new typed-wrapper pattern itself.
- New error `M3LSQSOperationError` (`ERR_SQS_OPERATION`), registered in
  `M3L_ERROR_CODES` alphabetically. `receive`/`purgeQueue` are deliberately
  **not** retried (long-poll absorbs transient emptiness;
  `PurgeQueueInProgress` is a 60s business cooldown, not a transient fault).
- Landed via `scaffolding-submodules` → `implementing-submodules` (contract →
  RED → GREEN → review fan-out: `code-reviewer`, `spec-conformance-reviewer`,
  `security-reviewer` plus a mandatory adversarial refute pass,
  `type-design-analyzer`, `silent-failure-hunter`).

## Outcome

Shipped on branch `feat/aws-sqs-operations` as PR #119: the `aws/sqs`
submodule, `docs/adr/0026-sqs-operations-wrapper.md`, and the Zone A
widening in `eslint.config.js`. 28 tests in `tests/sqs.test.ts` (26 authored
in RED, 2 added during review-fix — a `purgeQueue` not-retried regression and
an orphaned-`Failed[].Id` guard caught by `silent-failure-hunter`).
`client.ts` coverage cleared the submodule's 80% gate (93.24%
statements/84.48% branches/94.44% functions/92.95% lines). The review fan-out
also incidentally caught and fixed a latent bug in `bin/lib/count-sites.mjs`
itself: its "N of 22" denominator was hardcoded rather than derived, so
landing the 23rd submodule would otherwise have produced self-consistently
wrong "23 of 22" text across multiple docs.

Full narrative (the pivot from `sqs-etl` scaffolding, review verdicts,
divergences, and lessons) is recorded in
[`docs/logs/2026-07-13-aws-sqs.md`](../../logs/2026-07-13-aws-sqs.md). The
design decision itself is recorded in
[`docs/adr/0026-sqs-operations-wrapper.md`](../../adr/0026-sqs-operations-wrapper.md).
`sqs-etl` resumed and rebased onto this module afterward, consuming
`script.aws.clients.sqsOperations` with no `@aws-sdk/client-sqs` dependency.
