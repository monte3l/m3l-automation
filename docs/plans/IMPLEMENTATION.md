## AWS getter reality

getters are wrapped in a script-consumable `aws/<service>` operations
submodule today, and which are still raw SDK-client getters scripts are
banned from consuming directly (ADR-0027/ADR-0029). **Check this table
before scoping or starting any AWS-consumer-script item** — never infer
wrapper status from a getter's name/existence alone. 7 of 7 AWS scripts
shipped so far wrongly assumed "existing getter ✓" and had to pause to build
the wrapper first (see W3/W4 below). One getter = one row (ADR-0024
row-locality): the PR that ships a getter's `aws/<service>` wrapper flips
that row from raw → wrapped in the same commit.

## Consumer fleet (W2–W5)

Each script is one package under `scripts/<name>/`, scaffolded via
`pnpm scaffold:script`, built through the `scaffolding-scripts` →
`implementing-scripts` pipeline. A script lands as **one PR** only when every
`AWSClientProvider` getter it needs is already wrapped (see the
[AWS getter reality table](#aws-getter-reality) above); every AWS script to
date that needed a still-raw getter landed as a **two-PR chain** instead (the
`aws/<service>` wrapper submodule first, then the script) — check the table
before assuming 1-PR scope. Full per-family detail (config params, steps,
scale requirements §1.1–1.5) is in the archived
[`2026-07-09-consumer-scripts-implementation-plan.md`](./archive/2026-07-09-consumer-scripts-implementation-plan.md)
and its [roadmap](./archive/2026-07-09-consumer-scripts-roadmap.md).

### W2 — scale-hardened (P1, done)

- **`dynamodb-crud`** — **done.** `get/put/update/delete/query/scan/batch-write/batch-delete/export/import`; parallel `Scan` segments, `LastEvaluatedKey` page-loop, persisted checkpoints + `--resume` (keyed to a `runName` config parameter, not the per-invocation correlation id), 25-item batch chunks with a real retriable-classifier composition (`Core.combineClassifiers`) driving `UnprocessedItems` retry, streamed JSONL, the `aws/dynamodb` high-level operations (not raw `dynamoDBDocument`/SDK commands — the script never imports `@aws-sdk/*`) throughout, destructive gate. 83 tests; 3-reviewer fan-out (code-reviewer, security-reviewer, silent-failure-hunter) found and fixed 1 critical + 2 must-fix + 4 should-fix across two fix rounds; adversarial security refute pass confirmed clean. `aws/dynamodb` is recorded as wrapped in the [AWS getter reality table](#aws-getter-reality).
- **`cloudwatch-logs-insights`** — **done.** Queries CloudWatch Logs Insights over a time range split into fixed windows (to stay under the 10k-row `GetQueryResults` cap), exports JSON/CSV. Composes with `AWS.M3LLogsInsightsClient` (`aws/cloudwatch-logs-insights`, ADR-0027) via `startQuery()`+`awaitResults()` decomposition — never `runQuery()` — so an in-flight `queryId` can be checkpointed before the poll; the script never imports `@aws-sdk/*` directly. Checkpointed resume accumulates fetched rows in the checkpoint itself (the library's list exporters can only write a whole array, not append), writing the output file once at the end of the run; a terminal query failure aborts the whole run and leaves the checkpoint intact. 66 tests; 3-reviewer fan-out (code-reviewer, security-reviewer, silent-failure-hunter) found and fixed 1 must-fix + 6 should-fix across two fix rounds. `aws/cloudwatch-logs-insights` is recorded as wrapped in the AWS getter reality table.
- **`sqs-etl`** — **done**. `dump/send/redrive/delete/purge/transform` via `AWS.M3LSQSOperations` (`aws/sqs`, ADR-0026) — no direct `@aws-sdk/client-sqs` dependency. Long-poll receive loop (10/batch, budget-capped to `batchSize`), streamed JSONL, `sendBatch`/`deleteBatch` batching with `failed.jsonl` re-drive, shared destructive-gate confirmation (bypassable via `yes`), `transform` reusing `json-etl`'s filter grammar locally (a W5 promotion candidate once a 3rd script needs it). 102 tests. Checkpoint/resume and client-side rate capping deferred (see the script's contract page). `aws/sqs` is recorded as wrapped in the AWS getter reality table.

### W3 — control-plane CRUD (P1, `s3-objects`/`lambda-ops`/`eventbridge-schedules` done; remainder pending)

- **`s3-objects`** — **done.** `list/describe/get/put/copy/delete/delete-batch` over the `aws/s3` typed operations wrapper (ADR-0033; see [AWS getter reality table](#aws-getter-reality) above — s3 is now wrapped) — **not** the raw `s3` getter as this wave's original "existing getters ✓" premise assumed. Discovered during the `starting-work` gate before any code was written: dispatching a single S3 operation requires constructing an `@aws-sdk/client-s3` command object, which ADR-0027/ADR-0029 forbid inside `scripts/*/src`, so `aws/s3` shipped first as its own PR (`docs/logs/2026-07-18-aws-s3.md`). Destructive gate covers `put`/`copy`/`delete`/`delete-batch`; `delete-batch` chunks into ≤1000-key groups (S3's own `DeleteObjects` cap), validates each key record before a destructive call, and surfaces partial progress (`{ deleted, priorErrorCount }`) if a chunk's SDK call fails outright rather than losing it. 102 tests; 3-reviewer fan-out (code-reviewer, security-reviewer, silent-failure-hunter) converged independently on the same core bug (unwrapped local fs errors) plus 2 more must-fix findings, all closed in one fix round.
- **`lambda-ops`** — **done.** `list`/`describe`/`invoke`/`create`/`update-code`/`update-configuration`/`delete` over `AWS.M3LLambdaOperations` (`aws/lambda`, built mid-implementation this item since the raw `lambda` getter alone was not a consumable operation surface — the same correction `s3-objects` needed) — now recorded as wrapped in the [AWS getter reality table](#aws-getter-reality). Every operation except `list`/`describe` is confirm-gated (bypassable via `yes`); `invoke`'s function-level error (`functionError` populated, distinct from a `.send()`-level rejection) is turned into a non-zero exit (`ERR_LAMBDA_OPS_FUNCTION_ERROR`) only after persisting `output`, so the diagnostic payload survives on disk. 78 tests; 3-reviewer fan-out (code-reviewer, security-reviewer, silent-failure-hunter) found 2 must-fix + 2 should-fix (silent-failure-hunter returned a clean PASS), all fixed in one round.
- **`eventbridge-schedules`** — **done.** `list/describe/create/update/delete/enable/disable` via `AWS.M3LEventBridgeOperations` (`aws/eventbridge`, ADR-0027) — no direct `@aws-sdk/client-eventbridge` dependency; scoped to EventBridge **rules**, not the separate Scheduler service (see `docs/reference/aws/eventbridge.md`). `create`/`update` share a `putRuleStep` internal helper (both drive the same `PutRule` upsert) and may attach targets in the same call via a `targets` JSON config field, calling `putTargets` and logging (never throwing on) per-entry failures. Dynamic-import dispatch (mirrors `api-gateway-client`) so `vi.mock` resolves per step; destructive gate before every mutating op, bypassable via `yes`. 117 tests across 12 files; 3-reviewer fan-out (code-reviewer, security-reviewer, silent-failure-hunter) found zero must-fix, 4 should-fix (duplicated config-read helpers consolidated into `steps/config-helpers.ts`, misleading "EventBridge Scheduler" doc-comment wording fixed, `targets` shape-validation error enriched with index/field context) — all applied and re-verified clean. `aws/eventbridge` is recorded as wrapped in the AWS getter reality table.
- **`ecs-ops`** — pending. Thin op-dispatch (`list`/`describe`/`create`/`update`/`delete` + `waitUntilStable`) over the shared W1/W2 skeleton, mutating ops confirm-gated, over the `ecs` getter. **Check the [AWS getter reality table](#aws-getter-reality) before starting — `ecs` is raw today; do not assume it is op-dispatch-sufficient** (`s3-objects`/`lambda-ops`/`eventbridge-schedules` all needed a new wrapper first).
- **`cloudformation-stacks`** — pending (name per ADR-0028; not `cfn-stacks`). Thin op-dispatch (`list`/`describe`/`create`/`update`/`delete` + stack-event streaming) over the shared skeleton, over the `cloudFormation` getter. **Check the AWS getter reality table before starting — `cloudFormation` is raw today.**
- **`codepipeline-ops`** — pending. Thin op-dispatch (`list`/`describe`/`create`/`update`/`delete` + `watch`) over the shared skeleton, over the `codePipeline` getter. **Check the AWS getter reality table before starting — `codePipeline` is raw today.**

## Maintenance

When a unit lands: flip its `Status` cell in the relevant table above (and in
[`ROADMAP.md`](../ROADMAP.md)), `git mv` any dated plan it completes into
[`archive/`](./archive/), and file any new friction surfaced by a work log as a
new row here (not left narrative-only in the log).

New AWS-consumer-script items must state getter reality (wrapped/raw) sourced
from the [AWS getter reality table](#aws-getter-reality), not asserted from
the getter's name/existence alone. When a new `aws/<service>` operations
wrapper submodule ships, flip its row in that table from raw → wrapped **in
the same PR** (row-locality, ADR-0024: one getter = one row).

**Row-locality rule** (ADR-0024): one item = one table row; a status change
touches only that row; no prose sentence may aggregate more than one item's
status. The tables above already follow this — a branch flipping one F-series
item's status edits exactly one cell, so two branches landing different items
never conflict on this file.
