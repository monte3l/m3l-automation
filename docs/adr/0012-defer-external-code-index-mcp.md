# 0012. Defer external code-index MCP server; use native LSP + generated catalog

- **Status:** Accepted
- **Date:** 2026-06-30
- **Deciders:** Enrico Lionello

## Context and problem statement

As `@m3l-automation/m3l-common` grows from 5 to 22 implemented submodules,
cold-start navigation (finding where a symbol is defined, discovering which
modules exist, mapping an export name to its source file) will require repeated
broad `grep` sweeps across an expanding codebase. Each sweep consumes context
tokens and slows down Explore/implementer agents that need to locate symbols
quickly.

The question is: should an external code-indexing MCP server be added to the
Claude Code toolchain to make these lookups faster and more token-efficient?

## Decision drivers

- **Minimal-deps ethos**: the repo is strictly Node/pnpm/ESM; importing a
  foreign toolchain (Python, Rust binaries) conflicts with ADR-0001 and
  increases environment-setup friction.
- **LSP value already delivered**: `typescript-lsp@claude-plugins-official` is
  already enabled in `.claude/settings.json`, providing symbol-level
  go-to-definition, find-references, and hover natively.
- **Codebase is currently small**: only 5 of 22 submodules are implemented.
  The marginal grep cost is low and does not yet justify an indexer.
- **Committed, CI-verified artifacts** are preferable to in-process server
  state: the existing `bin/check-*.mjs` pattern proves that zero-dependency
  generated artifacts can fill the navigation gap without runtime overhead.

## Considered options

1. **Serena MCP server** — Python 3.13 + `uv`-managed; provides symbol-level
   semantic search and code navigation via an LSP-backed index.
2. **codebase-memory-mcp** — Node.js MCP server that builds a text-chunk index
   from source files; written in TypeScript/Node, zero Python dependency.
3. **claude-context** — Node.js MCP server using embedding vectors for
   semantic search; requires an external embedding provider.
4. **Native LSP plugin + generated catalog/symbol map** — lean on the already-
   enabled `typescript-lsp` plugin for live symbol navigation; add committed
   `docs/reference/catalog.json` and `docs/reference/symbol-map.json` artifacts
   (generated by `bin/gen-reference-index.mjs`, gated by `bin/check-reference-index.mjs`)
   for cross-module lookup and cold-start discovery.

## Decision

We chose **option 4 — native LSP + generated catalog/symbol map** and deferred
all external MCP indexers.

### Why not Serena (option 1)

Serena's headline capability — symbol-level go-to-definition and
find-references — is already delivered by `typescript-lsp@claude-plugins-official`
in `.claude/settings.json`. The marginal value of Serena is therefore low,
while its cost is high: it requires Python 3.13 and `uv`, neither of which
exists anywhere in this repo. Adding a Python toolchain for a feature already
covered by an enabled plugin violates ADR-0001 (lean, coherent toolchain) and
introduces a cross-language setup step for every contributor.

### Why not codebase-memory-mcp (option 2)

`codebase-memory-mcp` (and similar Node-only indexers) provides a text-chunk
index without Python, but its marginal value over the existing LSP plugin is
modest: it offers keyword search over file contents, not typed symbol lookup.
With only 5 submodules implemented today, grep sweeps are cheap and the added
server complexity is not yet justified. Revisit if/when the remaining 17 modules
land and agent grep cost becomes material.

### Why not claude-context (option 3)

`claude-context` needs an embedding provider (e.g. OpenAI or a local model),
which introduces an external service dependency, an API key, and a semantic gap
between the indexed representation and the actual code. Overkill for a library
with 22 submodules.

### Why the generated catalog is better than a live server (option 4)

A committed `catalog.json` + `symbol-map.json` pair:

- Lives in the repo, versioned alongside the code it describes — never stale.
- Requires zero runtime infrastructure (no MCP server, no `npx` at
  session-start, no port, no process management).
- Is gated by `pnpm check:index` in CI, so drift is caught at merge time rather
  than at agent-query time.
- Auto-grows: each new submodule that ships a `*.provenance.json` sidecar
  automatically enriches the symbol map on the next `pnpm gen:index` run.
- Follows the established `bin/check-*.mjs` zero-dependency pattern.

## Consequences

- **Positive:**
  - No new toolchain language or runtime; the Node/pnpm/ESM stack stays uniform.
  - `docs/reference/catalog.json` gives agents an O(1) lookup path from symbol
    name to source file, replacing repeated grep sweeps.
  - `docs/reference/README.md` provides a human-scannable A–Z module index.
  - CI gate (`check:index`) prevents the catalog from drifting from the
    filesystem as new modules ship.
- **Negative / trade-offs:**
  - The catalog/symbol-map covers only committed state; it cannot reflect
    in-flight src/ changes made during a session (the LSP handles those).
  - Symbol coverage is limited to modules with `*.provenance.json` sidecars
    (currently 5); the remaining 17 appear in the catalog but with empty
    `symbols[]` until they ship.
  - If/when the codebase grows large enough that grep cost becomes material,
    this decision should be revisited and a Node-only indexer (option 2)
    considered.
- **Semver impact:** none — tooling and doc change only; no change to the
  public API.
