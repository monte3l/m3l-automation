# 0021. Post-1.0.0 direction: deepen the library first, then build consumers

- **Status:** Accepted
- **Date:** 2026-07-06
- **Deciders:** Enrico Lionello (maintainer); Claude (principal-architect
  strategy review)

## Context and problem statement

`@m3l-automation/m3l-common` shipped 1.0.0 with all 22 bootstrap submodules
implemented, reviewed, and documented (~20.5k LOC, 2,117 tests, 97.2%
statement coverage). The public contract is frozen at the three-entry
`exports` map (ADR-0004), the package is internal and unpublished with a
hand-managed version (ADR-0020), and the example consumer workspace was
removed before release (ADR-0019) — so the library currently has **zero real
consumers** and has never been exercised end-to-end outside its own tests.

With the bootstrap complete, the strategic question is: what should this
codebase become in its next iterations? A structured review mapped the whole
repository, drafted five candidate directions, narrowed them through a
maintainer interview, and feasibility-tested the four survivors against the
real code. This ADR records the direction chosen from that process; the
sequenced work lives in the companion roadmap
(`docs/plans/archive/2026-07-06-post-1.0-deepen-first-roadmap.md`).

## Decision drivers

- **Internal-only is settled.** The maintainer ratified ADR-0020's posture:
  no registry publishing, private or public, is on the table. Any direction
  premised on external adoption is out.
- **Solo consumer.** For the next 6–12 months the only consumers are the
  maintainer's own automation scripts.
- **No usage evidence yet.** Nothing has exercised the library end-to-end, so
  feature priorities cannot yet be grounded in observed pain.
- **Semver appetite: additive by default, 2.0 acceptable if earned.** Minor
  releases are preferred; a deliberate 2.0 is acceptable only if real usage
  exposes design errors.
- **Minimal runtime dependencies** and the frozen three-entry `exports` map
  remain non-negotiable; new symbols surface through the namespace barrels.
- **Focused-phase cadence.** Work lands in concentrated, bootstrap-style
  pushes driven by the hub-and-spoke TDD pipeline.
- **Platform reuse is wanted eventually.** The maintainer wants the repo's
  agent/skill/hook engineering system reused in other repos, but not at the
  expense of the library itself.

## Considered options

The interview dropped one direction outright: **external adoption /
publishing (D2)** — internal-only was ratified, reinforcing ADR-0020. The
four survivors were feasibility-tested against the codebase, then put to the
maintainer as four orderings:

1. **Consumer-first flywheel** (architect-recommended) — re-introduce an
   in-repo `scripts/*` consumer workspace first (S effort, unblocked,
   ADR-0019-compatible for _real_ scripts), and let observed usage drive
   which deepen/broaden features get built.
2. **Deepen first, consume later** — ship the five verified additive-minor
   library features now as a 1.1 wave (~11h estimated), then stand up
   consumers against the richer library.
3. **Dual-track** — in-repo consumers plus the minimal-path extraction of
   the engineering platform in parallel focused phases.
4. **Platform-first** — generalize the agent/skill/hook/worktree/provenance
   system now; treat the library as feature-complete until consumers appear.

Feasibility findings common to all options: the five deepen features
(registerable text extractors, polling/retry telemetry, correlation-ID
tracing, schema-time config validators, preset inheritance) are all additive
through existing seams — minor releases, `exports` map untouched. New
submodules (broaden) are mechanically additive but every candidate is gated
on consumer pull, and the Slack/webhook messaging transport is blocked on
`M3LHttpClient` gaining POST support. Platform extraction is viable at a
copy-and-parameterize tier (~1 day) but needs a second repo to port into.
A separate-repo consumer is blocked by the unpublished/internal posture.

## Decision

We chose **option 2 — deepen first, consume later**.

The next iteration ships the five feasibility-verified additive features as a
1.1 wave, in this scope:

1. **Registerable text extractors** — surface the existing
   `M3LTextExtractorRegistry.register()` seam and the `M3LTextExtractor`
   contract as a documented public extension point.
2. **Polling/retry telemetry** — optional event/logger wiring on `M3LPoller`
   and `M3LRetryRunner` (attempt, success, failure, exhaustion payloads).
3. **Correlation-ID tracing** — an optional per-run correlation ID threaded
   through `M3LScript` hook contexts, `M3LLogEvent`, and `M3LError` context.
4. **Schema-time config validators** — optional `validator` support on config
   parameters plus stock helpers (range, regex, predicate).
5. **Preset inheritance** — an `extends` mechanism in the script preset
   loader with cycle detection and a new registered error code (merge
   semantics to be settled in the roadmap's design gate before
   implementation).

Residuals ride along: coverage backfill for the known soft spots and the
ESLint dependency-direction zones already accepted in ADR-0009.

After the wave lands and the version is hand-bumped to 1.1.0, the **in-repo
consumer fleet follows as the immediate next iteration** (real automation
scripts in a restored `scripts/*` workspace — consistent with ADR-0019, which
removed a hollow example, not the principle of real consumers). Broadening
(new submodules such as messaging transports, an SSM config provider, or a
Lambda invoke wrapper) stays **gated on a named consumer call-site**, and
platform extraction stays **deferred until a second repo exists** to port
into.

The maintainer explicitly accepted the known risk of this ordering: the five
features are prioritized without usage evidence. Mitigations: each feature
was verified cheap and strictly additive against real seams (so a wrong bet
costs little and breaks nothing), and the consumer fleet follows immediately,
converting hypothetical priorities into observed ones for every iteration
after this wave.

## Consequences

- **Positive:** the first real consumers meet a richer, more observable
  library (tracing, telemetry, validation, extensibility); all work is
  additive-minor with the `exports` map untouched and `check:api` green
  throughout; the deferred directions are recorded with explicit gates
  instead of being lost.
- **Negative / trade-offs:** feature priorities are chosen without usage
  evidence — some of the five may see little use; the maintained public
  surface grows ahead of demand; consumer feedback (and any 2.0-worthy
  design lesson it might surface) arrives one iteration later than the
  consumer-first ordering would have delivered it.
- **Semver impact:** minor — the wave lands as 1.1.0 (hand-managed version
  per ADR-0020). No `exports`-map change; new symbols surface through the
  existing `core`/`aws` namespace barrels only.
